# 0014 - Voice-native CRM: inbound calls answered from real data

**Date:** 2026-07-19 · **Status:** SHIPPED (code, sandbox-verified) · **Owner:** the engineer (the human on the felt experience)

## The decision

The operator calls their own AgentPhone number and asks, out loud, "who do I
need to follow up with today" (or "what did my agent do while I was away",
"what's hot in my pipeline"). AgentPhone transcribes the request and POSTs it
to a new Scalar webhook; Scalar routes the intent to the real CRM, and returns
a short, spoken-friendly, grounded answer for AgentPhone to voice back. This is
the inbound half of the phone surface - `place_call` (outbound, shipped in
0004/existing MCP tools) already lets an agent call a lead; this lets the
*operator* call *Scalar*.

Built:

- **`src/lib/voice-intent.ts`** - the intent bridge. A no-LLM heuristic
  classifier in the same `{ id, purpose }` + regex shape as
  `src/lib/intent-router.ts`, scoped to four intents (`followups`, `pulse`,
  `pipeline_hot`, `search`) plus `unknown`. Each intent calls an existing,
  already-tenant-scoped op (`listDueFollowups`, `computePulse`, `listContacts`,
  `searchCrm` - nothing new added to the ops layer) and renders a short,
  plain-sentence, no-markdown summary with rounded counts and at most 3 named
  items - written to be read aloud, not rendered. Never throws: any op failure
  degrades to a safe spoken apology (console.warn'd), and empty results are
  never bragged as a false positive (mirrors The Pulse's own falsifier rule).
- **`src/app/api/webhooks/agentphone/route.ts`** - the inbound webhook (public;
  already covered by the existing `/api/webhooks(.*)` matcher in
  `src/proxy.ts`, no proxy change needed). Authenticates the caller (see
  below), defensively extracts a transcript from an payload whose real shape
  is unverified, calls `voiceIntent`, returns `{ speech }`, and logs an
  `Activity` (`kind: "call"`, `channel: "phone"`) for auditability - all
  wrapped so a live caller always gets *something* to hear, never a bare 500.
- **`src/app/api/settings/voice/route.ts`** + **`VoiceSettingsForm`** - a
  Settings card (mirroring `AgentPhoneKeyForm` / `WebhookUrl` styling) to turn
  voice on (blocked until AgentPhone is connected), see/copy the exact webhook
  URL to paste into AgentPhone, and rotate it.
- **`User.voiceEnabled` / `User.voiceInboundSecret`** - additive Prisma
  columns, mirrored into `prisma/supabase-setup.sql` via `ADD COLUMN IF NOT
  EXISTS` (that file's `users` table already predates several later columns -
  see the SQL comment; this follows the same additive pattern without
  attempting to fix that pre-existing drift).
- **`scripts/voice-smoke.mjs`** (`pnpm smoke:voice`) - exercises the live
  webhook with synthetic payloads (auth refusal, tenant data, malformed body)
  without ever placing a real call or touching AgentPhone's API. Reports "not
  configured" and exits cleanly when no key is set (verified in-sandbox).
- Tests: `tests/voice-intent.test.ts` (40 total with the webhook file) cover
  intent classification, grounded/empty-safe speech, and CRM-op routing;
  `tests/voice-webhook.test.ts` covers the security boundary (below) and
  defensive parsing of malformed/absent/wrong-typed payload fields.

## Caller authentication - the security-critical decision

The endpoint answers with real CRM data over a public HTTP route, so the
question is: **how do we know who is calling?**

Options considered:
1. **Match on the inbound phone number** against a number stored on `User`.
   Rejected: AgentPhone's payload shape for the inbound number is unverified,
   phone numbers are not secret and can be spoofed at the carrier/VoIP level,
   and no live account exists to confirm the field even arrives reliably.
2. **Trust an account/agent id in the JSON body.** Rejected outright: the body
   is untrusted input from an unverified third party. A test
   (`tests/voice-webhook.test.ts`: "a spoofed userId/accountId in the JSON
   body is never trusted for identity") proves the route ignores any such
   field.
3. **A per-user, high-entropy secret baked into the webhook URL itself**
   (`?key=<256-bit token>`), looked up by direct equality against a unique
   indexed `User.voiceInboundSecret` column. **Chosen.** This is a capability
   URL - the same pattern Stripe Connect, Zapier, and most third-party
   inbound-webhook receivers use for per-tenant endpoints. It needs no
   assumption about AgentPhone's payload shape, signing scheme, or number
   provenance at all: the secret is generated by Scalar, never derived from or
   dependent on anything AgentPhone sends, and rotates independently the
   moment the operator asks (Settings "New link" button).

Enforcement, in order, on every request:
- Rate-limited by IP (30/min) before touching the database.
- Missing or implausibly short `key` (< 32 chars) -> `401`, no DB query at
  all.
- `key` matches no `User.voiceInboundSecret` -> `401`.
- `key` matches a user, but `voiceEnabled` is false or `agentPhoneApiKey` is
  unset -> `501` ("voice not configured"), so disabling voice or
  disconnecting AgentPhone immediately invalidates a still-technically-valid
  key without deleting it.
- Only past all four checks does `voiceIntent(user.id, transcript)` run,
  scoped to that resolved `user.id` and nothing else the payload claims.

Tenant isolation is proven directly: `tests/voice-webhook.test.ts` sends the
same wrong-account claim under two different valid keys and asserts each only
ever sees its own user's data, never the other's, and that `Activity.userId`
on the audit log is always the resolved identity, never anything from the
body.

**If the key cannot be matched, the endpoint refuses. It never falls back to
"the only configured user," a default account, or any other guess.**

## Debts owed to reality

This is the honest list of what is **asserted/reasoned in-sandbox**, not yet
**observed** against a live account - do not treat this feature as fully
proven until these are closed:

1. **AgentPhone's real inbound-call payload shape is unverified.**
   `extractTranscript()` in the webhook route guesses at plausible field names
   (`text`, `transcript`, `transcription`, `message`, `query`, `input`,
   `utterance`, `speech`, `prompt`, `request`, and a couple of nested `call.*`
   / `data.*` shapes). If the real field name differs, every call gets the
   generic "I didn't catch that" fallback instead of a real answer - not a
   crash, but not useful either. `scripts/voice-smoke.mjs`'s header documents
   the exact steps to close this: connect a real account, make one real call,
   inspect the logged request, add the real field name to the candidate list.
2. **Whether AgentPhone signs its webhook requests at all** (e.g. an HMAC
   header) is unknown. The per-user secret alone is sufficient to authenticate
   (see above), but a signature check would add defense in depth against a
   leaked URL being replayed from a non-AgentPhone origin. Add it once
   AgentPhone's docs/behavior confirm a scheme exists.
3. **Whether AgentPhone actually reads the `speech` JSON field aloud**, and in
   what response envelope/status it expects, is unverified. If the real
   contract differs (a different key name, SSML markup, a required
   `Content-Type`, etc.), the response shape in `route.ts` needs a matching
   update.
4. **Inbound number provisioning** - how the operator gets an AgentPhone
   number that rings into this webhook at all - is entirely on the
   AgentPhone side and outside this change; Settings only shows the webhook
   URL to paste in, it does not provision a number.
5. **One live end-to-end round trip** (call in, hear a real follow-up list
   read back, confirm it matches the dashboard) has not happened. Everything
   above is TESTED-in-sandbox only until that call is made.

## Deferred (named, not silently dropped)

- **No LLM in the loop.** Classification is a heuristic regex router (matching
  `intent-router.ts`'s own no-LLM pattern), not an OpenAI call - deliberately,
  to avoid a second unverified dependency (voice + LLM) stacked on this cycle.
  A future pass could add an LLM fallback for phrasing the heuristic misses,
  the same way `intent-router.ts` notes an LLM can sit in front of its own
  heuristic.
- **No webhook signature verification** beyond the per-user secret (see debt
  #2) - correct to defer until AgentPhone's real signing behavior, if any, is
  known; adding a check for a signature scheme that does not exist would be
  security theater.
- **Voice is not currently metered/credited.** Unlike `place_call` (20
  credits), answering an inbound call costs Scalar nothing extra (no
  third-party API called), so it is free by design - flagged here as a
  founder call if usage-based pricing on voice minutes is wanted later.
- **The audit Activity is written but not yet surfaced in the UI.** Most
  voice intents (follow-ups, pulse, pipeline-hot) aren't about one specific
  contact or company, so the logged row has `contactId`/`entityId` both null
  and won't appear on any per-contact activity trail (`listActivities`
  requires one of those ids) or on the dashboard (no global activity feed
  exists yet). The row is real and queryable directly, just not yet given a
  home in the product - a natural follow-up would be a small "recent voice
  calls" list in Settings or on the dashboard.
