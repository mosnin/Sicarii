# MCP Server Audit - 2026-07-11

Scope: `src/app/api/mcp/[transport]/route.ts` (45 tools before this cycle, 48
after) plus every downstream op, provider client, auth path, and the AgentPhone
UI chain. Follow-up to `mcp-audit-2026-06-08.md`.

**Headline:** tenant isolation is clean across all tools (no cross-tenant
read/write found anywhere, including the feared sync_call logId path). Metering
has no double-charges; x402 grants are exactly-once. The real findings were one
SSRF, a silently-lying `remember` tool, an unvalidated `recordingUrl` rendered
as a raw anchor, and a set of policy/validation gaps. Everything marked FIXED
below was remediated in the same PR that added the social channels.

## Findings and status

### P1 (security / correctness) - all FIXED this cycle

- **SSRF in `detect_tech`** - `detectSiteTech` fetched agent-controlled
  `entity.website` with no guard while a purpose-built guard existed
  (`src/lib/ssrf.ts`) and was used for webhooks. An agent could probe internal
  hosts via `update_entity {website: "http://10.x.x.x/..."}` + `detect_tech`.
  **FIXED:** now routed through `safeHttpUrl` + `resolvesToPublicIp`.
- **`remember` lied on failure** - `storeMemory` silently no-ops without
  OPENAI_API_KEY or on insert failure, and the tool returned `{remembered: true}`
  unconditionally; an autonomous agent permanently lost context with a success
  receipt. **FIXED:** `storeMemory` returns a boolean; the tool reports
  `remembered: false` with a reason.
- **`log_call.recordingUrl` unvalidated + rendered as raw href** - accepted any
  string (incl. `javascript:`) and the calls card rendered it directly.
  **FIXED:** zod now requires a well-formed `https://` URL, and the renderer
  additionally refuses non-https values (defense in depth).

### P2 (policy / correctness polish) - FIXED this cycle

- `search_web` charged 2 credits even on zero results, violating the
  never-charge-a-miss policy every other metered tool follows. **FIXED.**
- `enrich_entity` debited before persisting the enrichment (a DB failure after
  debit charged without storing). **FIXED:** debit moved after the update.
- `get_contact` returned every ContactEmail ever saved (token blowout).
  **FIXED:** capped at 50 (and 50 social messages).
- Annotation drift: `extract_contact_details` and `google_search` carried
  `readOnlyHint: true` while debiting the meter. **FIXED.**
- Unbounded write inputs on create/update entity/contact and search queries.
  **FIXED:** `.max()` caps across the board.
- Entity `phone` column existed but was not settable via create/update entity
  zod. **FIXED.**
- `log_outreach.channel` only knew email/linkedin/phone. **FIXED:** now also
  x/instagram/facebook/other.

### Open (not fixed this cycle - tracked)

- **P2: provider API keys stored plaintext** - `User.agentMailApiKey` /
  `agentPhoneApiKey` are plaintext while Scalar's own keys are SHA-256 hashed.
  Needs encryption at rest (KMS or pgcrypto). Founder-adjacent (key rotation).
- **P2: `build_smart_segment` / `remember` / `recall` are unmetered and
  un-rate-limited** OpenAI spend (embedding up to ~200 rows per call). Pricing
  decision owed: add a credit cost or a `gated()` bucket.
- **P2: only ~12 of 48 tools are rate-limited at all**; rate limiting is durable
  only when Upstash is configured (in-memory per-instance otherwise, fails open).
- **P2: opaque provider failures** - Exa/Apify/Tavily/AgentPhone client errors
  collapse to "Internal error"; `verify_entity` conflates provider outage with
  "no record found". An agent cannot distinguish bad-key from outage.
- **P3: annotation backfill** on the original ~23 tools (still no annotations).
- **P3: plan pricing oddity** - business ($99 / 8,000 credits) is worse per
  credit than pro ($129 / 12,000); will confuse a cost-optimizing agent.
  Founder pricing call.
- **AgentPhone endpoint shapes remain unverified against the live API**
  (`src/lib/agentphone.ts` says so honestly); if `placeCall` returns an
  unexpected shape the call logs with no id and can never be synced. Owed to
  reality: one live call.
- **OAuth signing secret still falls back to CLERK_SECRET_KEY** (now with a loud
  production warning). Set OAUTH_SIGNING_SECRET in prod.

## Coverage gaps for an operating agent (confirmed, prioritized)

1. ~~Social profiles / social conversations~~ **CLOSED this cycle**
   (`find_socials`, `log_social_message`, `list_social_messages`, social fields
   on create/update_contact + source attribution).
2. Intent monitors: no MCP tools (create/list/pause/delete) - an agent cannot
   set up its own recurring discovery. REST-only.
3. Research schedules: same.
4. Email: no send tool, no live-inbox list; AgentMail is UI-read-only. ~~Saved
   context came back only via get_contact~~ **PARTIALLY CLOSED** (2026-07-18):
   `list_emails(contactId, limit)` gives a dedicated, paged read of a contact's
   email history, matching `list_social_messages`. Sending mail and a live
   inbox list are still open.
5. spawn_contacts / deep-report / analyze-site / bulk-enrich: REST-only.
6. Segment/pipeline cleanup: no update/delete segment, no remove-member, no
   delete pipeline - agents can create but never clean up Field objects.
7. ~~Provenance read: FieldProvenance powers the UI trust moat but no MCP tool
   exposes it.~~ **CLOSED** (2026-07-18): `get_provenance(recordType, recordId)`
   reuses `getProvenanceMap`'s userId-scoped ownership fence (FieldProvenance
   has no userId column of its own; the fence checks the parent contact/entity
   first) so the read can never leak cross-tenant.

## Auth + rate limiting

- API-key path rejects revoked keys correctly; `lastUsedAt` update is
  fire-and-forget (documented best-effort).
- OAuth: stateless HS256, 1h TTL (not revocable mid-lifetime by design); refresh
  rotation with atomic reuse-detection via RevokedToken is sound.
- Rate limiting durable only with Upstash configured; fails open on Redis errors.
